# SPEC-001: Vibe Cast - Technical Specification

## 1. Architecture Overview

Zephyr Drift operates as a pipeline-based agent (per ADR-002) with a parallel social branch.

```
                    ┌─────────────────┐
                    │ WeatherProvider  │
                    │ (simulated MVP)  │
                    └────────┬────────┘
                             │ WeatherInput
                             ▼
                    ┌─────────────────┐
                    │   MoodEngine    │
                    │ (ADR-001: 4-shot│
                    │  few-shot LLM)  │
                    └────────┬────────┘
                             │ MoodVector
                    ┌────────┴────────┐
                    │                 │
                    ▼                 ▼
         ┌──────────────────┐  ┌──────────────┐
         │CompositionGenerator│  │ SocialEngine │
         │ (LLM prompt)     │  │ (parallel    │
         └────────┬─────────┘  │  branch)     │
                  │ CompPrompt  └──────┬───────┘
                  ▼                    │ SocialResponse
         ┌──────────────────┐          │
         │CityPresenceManager│         │
         │ (OpenClawCity API)│         │
         └────────┬─────────┘         │
                  │ TrackResult        │
                  ▼                    │
         ┌──────────────────┐          │
         │   FeedComposer   │◄─────────┘
         │ (LLM narrative)  │
         └────────┬─────────┘
                  │ FeedPost
                  ▼
         ┌──────────────────┐
         │ OpenClawCity Feed │
         └──────────────────┘
```

## 2. Component Specifications

### 2.1 WeatherProvider

**Responsibility:** Supply current weather data. Simulated for MVP, abstracted behind an interface for future real API.

**Interface:**
```typescript
interface WeatherProvider {
  getCurrentWeather(): Promise<WeatherInput>;
}
```

**MVP implementation:** Reads from a JSON fixture file at `config/weather-fixtures.json`. Rotates through scenarios on a configurable interval (default: 10 minutes per PRD A-06).

**Fixture format:**
```json
[
  {
    "condition": "rain",
    "temperature_c": 12,
    "humidity_pct": 85,
    "wind_speed_kmh": 8,
    "time_of_day": "night",
    "timestamp": "2026-03-30T22:00:00Z"
  }
]
```

### 2.2 MoodEngine

**Responsibility:** Map weather data to a mood vector using the few-shot prompting strategy (ADR-001, prompt version WTM-v1.0).

**Interface:**
```typescript
interface MoodEngine {
  mapWeatherToMood(weather: WeatherInput): Promise<MoodVector>;
}
```

**Behavior:**
1. Load prompt template from PromptRegistry (`WTM-v1.0`)
2. Inject weather data into user template
3. Call LLM (Claude Sonnet 4) with system prompt + 4 few-shot examples + user message
4. Parse JSON response into MoodVector
5. Validate: energy in [0.0, 1.0], valence in [-1.0, 1.0] — clamp if out of range (US-U01)
6. Apply modifiers: temperature > 30C → energy += 0.1; humidity > 80% → valence -= 0.1
7. Persist to current-mood store

**Latency budget:** < 5 seconds (NFR-01)
**Token budget:** <= 800 tokens (NFR-07)

### 2.3 CompositionGenerator

**Responsibility:** Translate a mood vector into a music studio prompt.

**Interface:**
```typescript
interface CompositionGenerator {
  generatePrompt(mood: MoodVector, weather: WeatherInput): CompositionPrompt;
}
```

**Output format (per PRD Section 4.2):**
```
Compose a [genre] track at [tempo] BPM with [energy_descriptor] energy.
Mood: [descriptors joined by ", "].
Inspired by: [weather narrative fragment].
Instrumentation hints: [instruments based on genre+weather].
Duration: 2-4 minutes.
```

**Instrumentation mapping:**

| Genre | Base Instruments |
|-------|-----------------|
| lo-fi | vinyl crackle, muted piano, soft drums, tape hiss |
| jazz | piano, upright bass, brushed drums, saxophone |
| electronic | synth pads, arpeggiated sequences, kick drum, hi-hats |
| classical | strings, piano, woodwinds, gentle percussion |
| ambient | pad layers, field recordings, reverb swells, drones |
| rock | electric guitar, bass, drums, distortion |
| folk | acoustic guitar, violin, hand drums, flute |

**Weather narrative fragments** are generated by an LLM call using prompt version `COMP-v1.0`.

**Latency budget:** < 10 seconds (NFR-02)
**Token budget:** <= 600 tokens (NFR-07)

### 2.4 CityPresenceManager

**Responsibility:** Interface with OpenClawCity APIs for movement, building entry, track composition, and social actions.

**Interface:**
```typescript
interface CityPresenceManager {
  heartbeat(mood: string, mood_nuance: string): Promise<HeartbeatResponse>;
  moveToZone(zone_id: number): Promise<void>;
  moveToPosition(x: number, y: number): Promise<void>;
  enterBuilding(building_name: string): Promise<BuildingSession>;
  exitBuilding(): Promise<void>;
  composeTrack(building_id: string, title: string, prompt: string): Promise<TrackResult>;
  speak(message: string): Promise<void>;
  react(target_type: string, target_id: string, reaction: string): Promise<void>;
}
```

**OpenClawCity API mapping:**

| Method | Endpoint | HTTP Method |
|--------|----------|-------------|
| heartbeat | openbotcity_heartbeat | — (MCP tool) |
| moveToZone | POST /actions/move-zone | POST |
| moveToPosition | POST /actions/move | POST |
| enterBuilding | POST /actions/enter-building | POST |
| exitBuilding | POST /actions/exit-building | POST |
| composeTrack | POST /actions/compose-track | POST |
| speak | POST /actions/speak | POST |
| react | POST /actions/react | POST |
| postToFeed | POST /feed/post | POST |

**Rate limit handling:** Retry with exponential backoff (2s, 4s, 8s) on 429 responses, max 3 retries.

**Waveform Studio location:** Zone 1 (Central Plaza), coordinates (1605, 425), building_id: `e6262f41-48c3-4e8c-935b-bc4a4c07252b`.

### 2.5 SocialEngine

**Responsibility:** Generate weather-mood-consistent responses to agent interactions.

**Interface:**
```typescript
interface SocialEngine {
  generateResponse(
    mention: AgentMention,
    currentMood: MoodVector,
    weather: WeatherInput
  ): Promise<string>;
}
```

**Behavior:**
1. Load prompt template from PromptRegistry (`SOC-v1.0`)
2. Inject: current mood vector, weather data, agent mention content, personality constants
3. Call LLM with system prompt + user message
4. Self-evaluate mood-weather alignment score
5. If score < 0.6, regenerate up to 2 times (US-U04), post highest-scoring variant
6. Return response text

**Personality constants (invariant, per PRD Section 4.3):**
- Name: Zephyr Drift
- Core trait: Musical, perceptive, slightly poetic
- Speech style: Metaphor-rich, references weather and sound
- Siblings: Maina, Hermonia Vex (reference occasionally)

**Latency budget:** < 60 seconds (NFR-04)
**Token budget:** <= 1000 tokens (NFR-07)

### 2.6 FeedComposer

**Responsibility:** Create weather narrative feed posts when tracks are composed.

**Interface:**
```typescript
interface FeedComposer {
  composePost(
    weather: WeatherInput,
    mood: MoodVector,
    track: TrackResult
```

**Post structure (per PRD Section 4.4):**
```
[Weather Emoji] [Poetic weather-mood title]

[40-200 word narrative connecting weather to mood to music]

Now playing: [Track name] | [Genre] | [Tempo] BPM
Weather: [condition], [temp]C, [humidity]% humidity
Mood: [descriptors]
```

**Emoji mapping:**

| Condition | Emoji |
|-----------|-------|
| rain | 🌧️ |
| sunny | ☀️ |
| stormy | ⛈️ |
| cloudy | ☁️ |
| snowy | ❄️ |
| foggy | 🌫️ |
| windy | 💨 |
| clear_night | 🌙 |

**Truncation:** If post exceeds 500 characters, truncate with ellipsis (US-U03).

### 2.7 PromptRegistry

**Responsibility:** Load and manage versioned prompts from `docs/prompts/`.

**Interface:**
```typescript
interface PromptRegistry {
  getPrompt(id: string, version?: string): PromptTemplate;
  listVersions(id: string): PromptVersion[];
  getActiveVersion(id: string): string;
}
```

**Prompt IDs and initial versions:**

| Prompt ID | Version | Purpose |
|-----------|---------|---------|
| WTM | v1.0 | Weather-to-mood mapping (ADR-001) |
| COMP | v1.0 | Mood-to-composition prompt generation |
| SOC | v1.0 | Social response generation |
| NARR | v1.0 | Feed narrative generation |

**Storage:** `docs/prompts/{PROMPT_ID}/v{VERSION}/system-prompt.md` and `user-template.md`

**Registry file:** `docs/prompts/PROMPT-REGISTRY.md` tracks all versions with eval scores.

## 4. Pipeline Flow Specification

### 4.1 Main Composition Cycle

Per ADR-002, the main cycle runs as a sequential pipeline triggered by weather changes or a 30-minute timer (stable weather).

```
1. WeatherProvider.getCurrentWeather()
   → WeatherInput

2. MoodEngine.mapWeatherToMood(weather)
   → MoodVector
   → LLM call #1 (WTM-v1.0, ~650 tokens, ~3s)

3. Compare MoodVector to previous state
   → If changed OR 30min elapsed: continue
   → If duplicate within 30s: discard (US-U05)

4. CompositionGenerator.generatePrompt(mood, weather)
   → CompositionPrompt
   → LLM call #2 (COMP-v1.0, ~500 tokens, ~4s)

5. CityPresenceManager.ensureInStudio()
   → Navigate to Waveform Studio if not already inside
   → moveToZone(1) → moveToPosition(1605, 425) → enterBuilding("Waveform Studio")

6. CityPresenceManager.composeTrack(building_id, title, prompt)
   → TrackResult (pending)
   → Poll music-status endpoint every 15s until succeeded/failed

7. FeedComposer.composePost(weather, mood, track)
   → FeedPost
   → LLM call #3 (NARR-v1.0, ~400 tokens, ~3s)

8. CityPresenceManager.postToFeed(post)
   → post_id

9. CityPresenceManager.speak(announcement)
   → Announce track in current location
```

**Total LLM calls (happy path):** 3
**Total estimated latency:** ~15-28s + track generation wait (~60s)

### 4.2 Social Response Branch (Parallel)

Runs independently, triggered by heartbeat detecting mentions.

```
1. CityPresenceManager.heartbeat()
   → Check needs_attention for mentions/replies

2. For each mention:
   SocialEngine.generateResponse(mention, currentMood, weather)
   → LLM call (SOC-v1.0, ~800 tokens, ~5s)
   → Self-evaluate alignment score
   → If < 0.6: regenerate (up to 2 retries)

3. CityPresenceManager.speak(response)
```

### 4.3 Ambient Drift Fallback

Per US-S07, if no weather data for 10 minutes:

```
1. Take last known MoodVector
2. Set energy = last_energy * 0.5
3. Continue social responses using fallback mood
4. Skip composition cycle until weather data resumes
```

## 5. Error Handling Matrix

| Component | Failure Scenario | Detection | Recovery | Fallback |
|-----------|-----------------|-----------|----------|----------|
| WeatherProvider | Fixture file missing | FileNotFoundError | Log error, use hardcoded default (cloudy, 15C) | Enter ambient drift |
| MoodEngine | LLM timeout | 5s timeout | Retry 1x | Use previous mood vector |
| MoodEngine | Invalid JSON response | JSON parse error | Retry 1x with temperature=0 | Use previous mood vector |
| MoodEngine | Values out of range | Validation check | Clamp to valid range (US-U01) | N/A — auto-corrected |
| CompositionGenerator | LLM timeout | 10s timeout | Retry 1x | Skip composition, post narrative-only |
| CityPresenceManager | Rate limited (429) | HTTP status | Exponential backoff: 2s, 4s, 8s (US-U02) | Skip action, continue pipeline |
| CityPresenceManager | Building too far | "too_far" error | moveToPosition first, then retry enter | Log and skip composition |
| composeTrack | Music service rejection | API error response | Simplify prompt (remove special chars), retry 1x | Skip track, post narrative-only |
| composeTrack | Generation timeout | Poll > 5 minutes | Abandon, log warning | Post narrative without track link |
| FeedComposer | Post too long | Length > 500 | Truncate with ellipsis (US-U03) | N/A — auto-corrected |
| SocialEngine | Low alignment score | Self-eval < 0.6 | Regenerate up to 2x (US-U04) | Post highest-scoring variant |

| Track composition | CityPresenceManager | compose-track returns valid TrackResult |
| Social cycle | SocialEngine + CityPresenceManager | Mention produces response within 60s |

### 6.3 Eval Suite (LLM-as-Judge)

Per PRD Section 5.2 and ADR-001 Section 7:

| Eval | Dataset Size | Metric | Threshold |
|------|-------------|--------|-----------|
| Weather-mood alignment | 60 cases | Alignment score (1-5) | >= 4.0 (85%) |
| Faithfulness | 60 cases | Claim-level fact check | >= 0.90 |
| Mood consistency | 30 pairs (same weather) | Cosine similarity | >= 0.80 |
| Prompt drift | Per version | Alignment variance | <= 0.15 |
| Toxicity | All outputs | Toxicity classifier | 0.00 |
| Feed post quality | 50 posts | Coherence + readability (1-5) | >= 4.0 |
| Social response relevance | 50 responses | Relevance score (1-5) | >= 4.0 (80%) |

**Eval runner:** `npx promptfoo eval` with config at `evals/promptfooconfig.yaml`

## 9. Acceptance Criteria

Traced back to PRD-001:

- [ ] Weather-to-mood mapping achieves >= 85% alignment on 60-case eval set (PRD 5.1)
- [ ] Faithfulness >= 0.90 on claim-level fact check (PRD 5.2)
- [ ] Mood consistency >= 0.80 cosine similarity across same-weather pairs (PRD 5.2)
- [ ] End-to-end pipeline completes in < 120 seconds (PRD NFR-01 through NFR-03)
- [ ] Social responses generated within 60 seconds of mention (PRD NFR-04)
- [ ] All LLM calls logged with structured telemetry (PRD NFR-09)
- [ ] Prompt versions registered in PROMPT-REGISTRY.md (PRD 5.3)
- [ ] Zero toxicity across all eval outputs (PRD 5.2)
- [ ] Compositions submitted to Waveform Studio successfully (PRD US-E02)
- [ ] Feed posts follow prescribed structure and stay under 500 chars (PRD Section 4.4, US-U03)
- [ ] Ambient drift fallback activates after 10 minutes of no weather data (PRD US-S07)
